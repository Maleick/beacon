# AutoShip Verification Pipeline

Invoked after an agent reports completion. Runs the full verify тЖТ simplify тЖТ verify тЖТ PR тЖТ monitor тЖТ cleanup pipeline.

## Step 0: Discover Test Command

Before verification, determine the repo's test command. Check in order:

1. `.autoship/config.json` тЖТ `test_command` field (operator override)
2. `package.json` тЖТ `scripts.test` тЖТ use `npm test` or `yarn test`
3. `Makefile` тЖТ `test` target тЖТ use `make test`
4. `Cargo.toml` тЖТ use `cargo test`
5. `pyproject.toml` / `setup.py` тЖТ use `pytest` or `python -m pytest`
6. `go.mod` тЖТ use `go test ./...`
7. If none found: skip test step, flag in reviewer prompt as "no test command discovered"

Cache the discovered command in `.autoship/state.json` under `test_command` so subsequent verifications skip discovery.

## Step 0.5: Pre-Verification Guards

Before spawning the reviewer, run these assertions. Any failure тЖТ immediate FAIL, no reviewer spawned.

**1. Path canonicalization** тАФ assert `AUTOSHIP_RESULT_PATH` is inside `WORKTREE_PATH`:

```bash
# Check existence first тАФ missing file is a distinct failure from symlink escape
if [[ ! -e "$AUTOSHIP_RESULT_PATH" ]]; then
  echo "VERDICT: FAIL тАФ AUTOSHIP_RESULT.md missing (agent did not write result file)"
  exit 1
fi
REAL_RESULT=$(realpath "$AUTOSHIP_RESULT_PATH" 2>/dev/null)
REAL_WORKTREE=$(realpath "$WORKTREE_PATH" 2>/dev/null)
if [[ -z "$REAL_RESULT" || -z "$REAL_WORKTREE" || "$REAL_RESULT" != "$REAL_WORKTREE"/* ]]; then
  echo "VERDICT: FAIL тАФ AUTOSHIP_RESULT_PATH is outside worktree (possible symlink escape)"
  # Mark issue blocked, skip to escalation
fi
```

This prevents a worker agent from writing `AUTOSHIP_RESULT.md` as a symlink pointing outside the worktree.

**2. Non-empty diff** тАФ assert the agent actually made changes:

```bash
DIFF_OUTPUT=$(git -C "$WORKTREE_PATH" diff main...HEAD 2>/dev/null)
if [[ -z "$DIFF_OUTPUT" ]]; then
  echo "VERDICT: FAIL тАФ git diff main...HEAD is empty. No changes were committed."
  # Mark issue blocked, skip to escalation
fi
```

An empty diff means the agent reported COMPLETE without committing anything. The reviewer **must not** be allowed to PASS an empty diff тАФ this guard is mandatory and cannot be overridden by the result file contents.

## Step 1: Initial Verification

Spawn the `reviewer` agent with these structured variables:

| Variable              | Value                                                     |
| --------------------- | --------------------------------------------------------- |
| `ISSUE_TITLE`         | The GitHub issue title                                    |
| `ISSUE_BODY`          | Full issue body                                           |
| `ACCEPTANCE_CRITERIA` | Parsed from issue or generated by Opus                    |
| `AUTOSHIP_RESULT_PATH`  | `.autoship/workspaces/<issue-key>/AUTOSHIP_RESULT.md`         |
| `WORKTREE_PATH`       | `.autoship/workspaces/<issue-key>`                          |
| `DIFF_COMMAND`        | `git -C .autoship/workspaces/<issue-key> diff main...HEAD`  |
| `TEST_COMMAND`        | The command discovered in Step 0 (or "none" if not found) |

The reviewer will execute `DIFF_COMMAND` and `TEST_COMMAND` itself тАФ do **not** pass pre-rendered output.

### AUTOSHIP_RESULT.md Expected Format

Agents are instructed to write this file. If missing, treat as FAIL. Expected structure:

```markdown
# Result: #<number> тАФ <title>

## Status: DONE | PARTIAL | STUCK

## Changes Made

- <file>: <what changed and why>

## Tests

- Test command: `<command>`
- Result: PASS | FAIL
- New tests added: yes/no

## Notes

<anything the reviewer should know>
```

The reviewer should cross-reference this self-report against the actual `git diff` and test output. Discrepancies (e.g., agent claims PASS but tests fail) тЖТ automatic FAIL.

Expect a structured verdict: PASS, FAIL, or LOW_CONFIDENCE.

### On FAIL

Report to Opus with failure reason. Opus decides:

- Re-dispatch to different tool (if attempts < 3)
- Mark as blocked (if all tools tried)
- Refine acceptance criteria and retry same tool

### On LOW_CONFIDENCE

Opus reviews the situation and makes the final call.

### On PASS

Continue to Step 2.

## Step 2: Simplify

Before simplification, create a rollback point:

```bash
cd .autoship/workspaces/<issue-key>
git tag autoship-pre-simplify-<issue-key>
```

Build the explicit file list before spawning:

```bash
SIMPLIFY_FILES=$(git -C .autoship/workspaces/<issue-key> diff --name-only main...HEAD)
```

Spawn a Sonnet agent in the worktree to run code simplification with the following constraints embedded in the prompt:

- **Scope**: "Simplify ONLY these files: [$SIMPLIFY_FILES]. Do not open, read, or modify any other files."
- Run the `code-simplifier` skill if available, otherwise instruct Sonnet directly
- Constraints: must not break tests, must not change behavior, must not alter public API

## Step 3: Re-Verify

Spawn another `reviewer` to confirm simplification preserved correctness.

- Run tests again in the worktree
- Verify diff is still aligned with acceptance criteria
- Check that no new files were introduced by simplification

### On FAIL (simplification broke something)

Roll back to pre-simplification state:

```bash
cd .autoship/workspaces/<issue-key>
git reset --hard autoship-pre-simplify-<issue-key>
git tag -d autoship-pre-simplify-<issue-key>
```

Continue to Step 4 with the pre-simplification code. Log that simplification was reverted in the PR body.

## Step 4: Create PR

```bash
cd .autoship/workspaces/<issue-key>
# Stage only files changed from the base branch тАФ avoids staging AUTOSHIP_PROMPT.md,
# .watcher.pid, debug files, and other unintended artifacts
CHANGED_FILES=$(git diff --name-only main...HEAD)
if [ -n "$CHANGED_FILES" ]; then
  git add $CHANGED_FILES
fi
# If CHANGED_FILES is empty, the agent already committed everything тАФ skip the add step
git commit -m "<issue-key>: <issue-title>

Closes #<issue-number>

Dispatched by AutoShip. Agent: <tool-name>. Attempt: <N>.
Verified by Sonnet reviewer. Tests: passing."

gh pr create \
  --title "<issue-key>: <issue-title>" \
  --body "## Summary\n<from AUTOSHIP_RESULT.md>\n\n## Verification\n- Tests: passing\n- Reviewer: PASS\n- Simplified: yes/no\n\nCloses #<number>\n\nDispatched by [AutoShip](https://github.com/Maleick/AutoShip)" \
  --label autoship
```

## Step 5: Monitor PR

Spawn the `monitor` agent to watch:

- CI check status
- Automated review comments (Copilot, etc.)
- Merge conflicts

### If review comments found

Dispatch a Sonnet agent to address them in the worktree, push fixes to the PR branch.

### Merge decision

- Simple issues: `gh pr merge --squash --auto` immediately after CI passes
- Medium/Complex: Sonnet code review pass required before merge

## Step 6: Cleanup (after merge)

```bash
git worktree remove .autoship/workspaces/<issue-key> --force
gh issue edit <number> --remove-label autoship:in-progress
```

Update `.autoship/state.json`: mark issue as `merged` and increment stats:

```bash
hooks/update-state.sh set-merged <issue-id>
```
